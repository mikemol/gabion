# in/in-28.md
## The `in_step` Template as a Normative Protocol Artifact

### Purpose
This step installs a **canonical template** for future `in/` documents (`doc_role: in_step`) and elevates that template from a stylistic convention to a **normative protocol artifact**. The template constrains structure, obligations, artifact contracts, and auditability, ensuring future steps are mechanically comparable, drift-resistant, and evidence-aligned.

### Non-goals
- This step does **not** define new system semantics.
- This step does **not** mandate specific content for future steps, only structure and obligations.
- This step does **not** retroactively rewrite existing `in-23`–`in-27` documents.

---

## Status
- **This step installs:** a required structural schema for `doc_role: in_step`.
- **This step depends on:** the stabilized pattern emergent in `in-23`–`in-27`.
- **This step unblocks:** mechanically checkable documentation audits; automated doc drift detection.
- **Failure mode if absent:** future `in/` steps diverge structurally, weakening review, audit, and dependency reasoning.

---

## Definitions

- **in_step:** A normative, staged protocol document that introduces obligations, artifacts, or gates.
- **Template (normative):** A required structural skeleton whose omission or violation is an error, not a preference.
- **Face / Kit / Solver:** The canonical tri-partite decomposition binding obligations to implementation and discharge.

---

## Face / Kit / Solver

### Face (Obligations)

- **F1:** Every `doc_role: in_step` document shall conform to the canonical template structure.
  - **Witness hook:** Presence of required sections; frontmatter keys.
- **F2:** Each such document shall state explicit obligations and how they are discharged.
  - **Witness hook:** Face section with numbered obligations.
- **F3:** Each such document shall define explicit artifact contracts where artifacts are introduced.
  - **Witness hook:** Artifact Contract section.
- **F4:** Each such document shall be drift-resistant.
  - **Witness hook:** `doc_requires`, `doc_reviewed_as_of`, `doc_change_protocol`.

### Kit (Existing machinery)

- **Docs:** `in/in-23.md` … `in/in-27.md` establish the emergent pattern.
- **Policy:** `POLICY_SEED.md` §6 governs change control.
- **Repo structure:** `in/` vs `out/` separation already enforces staging.
- **CI:** Existing audits can be extended to validate structure.

### Solver (Implementation plan)

1. Install the canonical template text (this document) as the reference.
2. Require all future `doc_role: in_step` documents to conform.
3. Optionally introduce a lightweight structural audit:
   - validate frontmatter keys
   - validate required section headers
4. Reject or quarantine non-conforming `in_step` documents.

## Artifact Contract

### Required artifacts

#### `in/in-XX.md` (future steps)
- **Producer:** human authors
- **Consumer:** reviewers, CI audits, future steps
- **Schema:** markdown + required YAML frontmatter + required sections
- **Determinism:** section ordering is fixed
- **Update rule:** revisions bump `doc_revision`; structural changes require review

### Provenance
- Each document records:
  - dependency pins
  - authority level
  - allowed erasures
- “Latest” is never implicit; dependency drift must be explicit.

---

## Admissibility

- **Admissible:**
  - Adding new `in_step` documents that conform to the template
  - Revising existing documents with `doc_revision` increment
- **Inadmissible:**
  - Skipping required sections
  - Introducing obligations without discharge paths
  - Omitting artifact contracts where artifacts are introduced
- **Ambiguity policy:** content may vary; structure may not.
- **Non-explosion policy:** structural violations are reported, not silently fixed.

---

## Checks as Lemmas

- **Lemma L1 (discharges F1):** Structural audit verifies frontmatter + section presence.
- **Lemma L2 (discharges F2):** Face section contains numbered obligations.
- **Lemma L3 (discharges F3):** Artifact Contract present when artifacts named.
- **Lemma L4 (discharges F4):** Dependency pins and change protocol present.

## Success Criteria

- [ ] A new `in/in-29.md` can be written mechanically from the template.
- [ ] Reviewers can diff obligations and artifacts across steps structurally.
- [ ] CI (manual or automated) can detect missing required sections.
- [ ] Drift in dependencies is explicit and reviewable.

## Failure Modes and Diagnostics

- **FM1:** Missing frontmatter key → check YAML header against template.
- **FM2:** Obligations stated informally → enforce numbered Face entries.
- **FM3:** Artifacts mentioned but unspecified → add Artifact Contract section.
- **FM4:** Silent semantic edits → require `doc_revision` bump.
