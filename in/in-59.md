# in/in-59.md
## GitHub Issue Content Ingestion into the Planning Substrate Invariant Graph

### Purpose

Ingest GitHub issue content — body, comments, open/closed state — into the planning
substrate invariant graph as a first-class artifact ingress signal, on the same footing
as docflow compliance, controller drift, and test evidence. This makes issue-level
acceptance criteria and discussion context locally queryable, visible to the project-manager
view, and present in context-compacted LLM sessions.

### Non-goals

- Write-back: updating GH issue body from registry/touchpoint declarations.
- Milestone, project board, or PR ingestion.
- Replacing `sppf_checklist.md` with invariant-graph equivalents.
- Changing the outbound `sppf_sync_core.py` path (commit trailers → issue comments/labels).

---

## Status

- **This step installs:** a `github_issue` artifact ingress adapter in
  `invariant_graph.py`, a snapshot script (`scripts/github/issue_snapshot.py`),
  and optional GH-state validation in `sppf_status_audit.py`.
- **This step depends on:** the existing artifact ingress framework in
  `src/gabion/tooling/policy_substrate/invariant_graph.py` and the `gh` CLI
  (already a hard dependency via `sppf_sync_core.py`).
- **This step unblocks:** governance DSL queries against issue content; checklist/issue
  state consistency validation; project-manager view enrichment with acceptance criteria;
  reliable planning context in context-compacted LLM sessions.
- **Failure mode if absent:** GitHub issue content remains invisible to local tooling;
  the planning substrate cannot reason about issue state; planning context in GH
  discussions is lost under context compaction.

## Definitions

- **Issue snapshot:** a locally cached JSON array of GitHub issue records including body
  and comments, produced by `scripts/github/issue_snapshot.py` and written to
  `out/github_issues_snapshot.json`.
- **`github_issue` node:** a new node type in the invariant graph representing one GH
  issue, carrying `number`, `title`, `state`, `labels`, `body_text`, `comment_text`,
  `created_at`, `updated_at`.
- **Graceful degradation:** invariant graph builds normally when snapshot is absent; no
  existing graph semantics change.
- **State drift:** a checklist item marked `[x]` (done) whose linked GH issue is OPEN, or
  a `[~]`/`[ ]` item whose linked issue is CLOSED.

---

## Face / Kit / Solver

### Face (Obligations)

- **F59-1:** The snapshot script is idempotent and produces a deterministic output
  schema (additive over `gh issue list --json` fields).
- **F59-2:** The `github_issue` ingress adapter is optional — absent snapshot → no
  graph change; present snapshot → issue nodes lifted into graph.
- **F59-3:** CI snapshot step uses `continue-on-error: true` so GH API outages do not
  block the build.
- **F59-4:** State-drift warnings from `sppf_status_audit.py --gh-snapshot` are
  informative (not hard failures) while the snapshot matures.
- **F59-5:** Edge linkage between existing graph nodes and `github_issue` nodes uses
  whichever identifier source is structurally present — sppf_checklist `(GH-###)`
  annotations, commit-history git-state artifact, or touchpoint metadata — without
  requiring new mandatory fields on registry touchpoints.

### Kit (Existing machinery)

- Artifact ingress framework: `src/gabion/tooling/policy_substrate/invariant_graph.py`
- Runtime entrypoint: `src/gabion/tooling/runtime/invariant_graph.py`
- `gh` CLI: already used by `src/gabion/tooling/sppf/sync_core.py`
- `_load_issues_json()` hook in `src/gabion_governance/governance_audit_impl.py`
  (handles basic `gh issue list --json` fields; can be extended with body/comments)
- `sppf_checklist.md` `(GH-###)` parsing in `_build_sppf_dependency_graph()`
  (candidate linkage source for F59-5)

### Solver (Implementation plan)

1. Create `scripts/github/issue_snapshot.py` — `gh issue list` + per-issue
   `gh issue view --json comments`; write `out/github_issues_snapshot.json`.
2. Add `github_issue` ingress adapter in `invariant_graph.py`; build `references`
   edges from sppf_checklist linkage as initial source (F59-5).
3. Expose `--gh-snapshot` flag on `invariant_graph` CLI entrypoint.
4. Add `--gh-snapshot` axis to `sppf_status_audit.py` (warnings only).
5. Add CI snapshot step; pass snapshot to invariant graph build.
6. Update `CONTRIBUTING.md` correction-unit stack.
7. Create `docs/github_issue_ingestion.md` governance contract doc.

## Artifact Contract

### `out/github_issues_snapshot.json`

```json
[
  {
    "number": 87,
    "title": "CLI/server report emission + LSP timeout robustness",
    "state": "OPEN",
    "labels": [{"name": "enhancement"}, {"name": "status/backlog"}],
    "body": "## Summary\n...",
    "comments": [{"author": "...", "body": "...", "createdAt": "..."}],
    "milestone": null,
    "createdAt": "2024-01-01T00:00:00Z",
    "updatedAt": "2024-01-02T00:00:00Z"
  }
]
```

- **Determinism:** field set is fixed; comment ordering is chronological.
- **Provenance:** fetch timestamp recorded in a top-level `_meta` field.

### `artifacts/out/invariant_graph.json` (extended)

New node type in existing graph:

```json
{
  "type": "github_issue",
  "number": 87,
  "title": "...",
  "state": "OPEN",
  "labels": ["enhancement", "status/backlog"],
  "body_text": "...",
  "comment_text": "...",
  "created_at": "...",
  "updated_at": "..."
}
```

## Admissibility

- **Admissible:** snapshot absent → graph builds without GH nodes (graceful degradation).
- **Admissible:** snapshot present → GH nodes ingested; existing node semantics unchanged.
- **Inadmissible:** snapshot errors that propagate as hard graph-build failures.
- **Inadmissible:** required mandatory `GH-{number}` fields on existing touchpoint definitions.
- **Ambiguity policy:** state-drift warnings are emitted explicitly; silence is not success.

---

## Checks as Lemmas

- **Lemma L59-1:** snapshot schema is additive over `gh issue list --json` and stable
  across re-runs for the same issue set.
- **Lemma L59-2:** invariant graph builds identically with snapshot absent as without
  snapshot (no side effects from missing optional input).
- **Lemma L59-3:** `github_issue` nodes carry no identity that collides with existing
  node types.
- **Lemma L59-4:** state-drift warnings are reproducible from snapshot + checklist alone.

---

## Success Criteria

- [ ] `out/github_issues_snapshot.json` contains `body` and `comments` fields for each
  issue.
- [ ] `artifacts/out/invariant_graph.json` includes `github_issue` nodes when snapshot
  is present; builds cleanly without snapshot.
- [ ] `sppf_status_audit.py --gh-snapshot` emits warnings for state drift; no hard
  failures.
- [ ] CI snapshot step uses `continue-on-error: true`; build passes on GH API outage.
- [ ] `docs/github_issue_ingestion.md` passes `docflow --fail-on-violations`.

---

## Failure Modes and Diagnostics

- **FM59-1:** GH API rate-limit or outage → snapshot fetch fails; `continue-on-error`
  prevents CI block; graph degrades gracefully.
- **FM59-2:** Schema drift in `gh issue list --json` output → snapshot parser returns
  partial records; diagnostic logged; no hard failure.
- **FM59-3:** Checklist `(GH-###)` annotation absent for a known issue → no edge created;
  issue node still present and queryable without linkage.
- **FM59-4:** Large comment volume → `comment_text` concatenation large; no functional
  impact (stored as string, not indexed).
